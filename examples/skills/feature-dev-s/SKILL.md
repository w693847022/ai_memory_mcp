# Feature 开发技能（拆分版）

## ⚠️ 重要指令

**不要并发执行步骤,后面步骤必须要前面步骤完全执行成功**

**DO NOT ENTER PLAN MODE** - 此技能要求直接执行，不进入计划模式。

**记忆查询**: 在需要回忆过往记录时（如查找相似功能、历史修复、相关笔记），可使用 `memory-query` 技能进行查询减少上下文。

**开发记录**: 阶段1后的每一步的关键结果记录到阶段1创建的[<feature_id>-开发记录]note中

**进度恢复**: 如果用户指定了已经记录功能和有该功能的开发记录的情况下，根据开发记录直接恢复之前上下文，然后跳到记录阶段向后执行。

---

## 流程概览

```
                                                                                                                                     
阶段1    →  阶段2    →  阶段3    →  阶段4    →  阶段5    →  阶段6    →  阶段7    →  阶段8    →  阶段9    →  阶段10   →  阶段11   →  阶段12   →  阶段13
项目确认 → 回忆规范 → 相关探索 → 需求澄清 → 需求确认 → 方案设计 → 方案选择 → 测试确认 → Git准备  → 代码修改 → 单元测试 → 全量测试 → 分支合并
                                                                                                       |        |           |
                                                                                                       ------<--失败---<-----
```

## 阶段 1: 项目确认

<!-- include: 项目确认.md -->
1. 建立feature
    - summary:<用户需求>
    - content:<用户需求>
2. 建立note命名[<feature_id>-开发记录]
---

## 阶段 2: 回忆开发所用规范

1. 使用技能`memory-std` 项目规范
2. 使用技能`memory-std` 代码开发相关规范

展示摘要：
```
[规范的summary] [规范的content（太长省略用...隐藏后续内容）]
[规范的summary] [规范的content（太长省略用...隐藏后续内容）]
...
```

---

## 阶段 3: 相关性探索

1. 相关功能探索
    1) 回忆相关项目功能
    2) 确认相关功能涉及代码范围
    3) 探索这些范围代码
       - 确认需求新旧界限（新功能添加还是旧功能优化）
           - 新功能是新到什么地步，数据操作链路需要如何新增
               - 需要的基础脚手架有没有
               - 现有框架引用库的情况下能不能轻松实现
               - 需要不需要引用新的库
           - 旧功能优化（已有的数据操作链路完全支持）
               - 确认使用的共用函数和类
               - 确认使用的外部库
    4) 从而弄清楚当前需求基础背景

## 阶段 8: 测试确认

1. 使用技能`memory-std` 测试规范
```
展示摘要：
[规范的summary] [规范的content（太长省略用...隐藏后续内容）]
[规范的summary] [规范的content（太长省略用...隐藏后续内容）]
...
```

2. 根据测试规范，确认的需求和选择的方案设计针对本次功能的单元测试文件。
3. 要求阐述清楚测试步骤，测试用例和边界情况。
4. 等待用户确认，如果用户要求修改则根据用户修改后回到本步骤第2项重新确认测试内容

---

## 阶段 9: Git 分支准备

<!-- include: Git分支准备.md -->

1. 调用 `git-branch` 技能执行分支准备
2. 按照 Git分支准备.md 中的流程创建开发分支

---

## 阶段 10: 代码修改

<!-- include: 代码实现.md -->

1. 按照确认的方案进行代码编写
2. 每完成一个功能模块后更新开发记录note
3. 完成后进入测试阶段

---
## 阶段 11: 本次需求的单元测试 

1. 根据之前确认的需求单元测试执行测试
2. 如果测试失败 **返回阶段10** 进行分析错误原因并修改代码
3. 循环超过3次仍然失败，暂停并请用户介入决策

---
## 阶段 12: 项目的全测试用例测试

1. 检查项目测试规范，如果有则执行项目全用例测试
2. 如果测试失败 **返回阶段10** 进行分析错误原因并修改代码
3. 循环超过3次仍然失败，暂停并请用户介入决策

---

## 阶段 13: Git 分支合并

<!-- include: Git分支合并.md -->

1. 调用 `git-branch` 技能执行分支合并
2. 合并完成后更新feature状态为completed
3. 整理开发记录note，补充合并信息
