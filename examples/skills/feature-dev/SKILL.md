# Feature 开发技能

## ⚠️ 重要指令

**DO NOT ENTER PLAN MODE** - 此技能要求直接执行，不进入计划模式。

此技能定义了完整的功能开发流程，包括需求澄清、方案设计、代码实现和记录。按照以下阶段直接执行即可，无需使用 EnterPlanMode。

所有的信息/代码搜索进入子代理模式进行

**记忆查询**: 在需要回忆过往记录时（如查找相似功能、历史修复、相关笔记），可使用 `memory-query` 技能进行查询。

---

结构化功能开发流程，从需求分析到代码实现，通过 Memory MCP 记录全过程。

---

## 流程概览

```
回忆规范 → 项目确认 → 相关性探索 → 需求澄清 → 需求确认 → 方案设计 → 方案选择 → Git分支准备 → 代码实现 → Git分支合并 → 完成记录
   │          │          │          │          │          │          │          │           │          │           │
   └─调用技能─→ 确定项目 ─→ Memory查询 ─→ 精准提问 ─→ Memory记录 ─→ 直接设计 ─→ 用户选择 ─→ 调用技能 ─→ Auto Accept ─→ 调用技能 ─→ 更新状态
```

## 阶段 0: 回忆项目规范

**目标**: 在开始功能开发前，回忆相关项目规范

**流程**:
1. 调用 `rememory-std` 技能获取项目规范
2. 根据功能类型选择合适的规范参数：
   - 代码功能 → `Skill: rememory-std, args: "代码规范"` 或 `"开发规范"`
   - API 功能 → `Skill: rememory-std, args: "API规范"`
   - 文档相关 → `Skill: rememory-std, args: "文档规范"`
   - 不确定类型 → `Skill: rememory-std` (获取所有规范)
3. 解析返回的 JSON 格式规范，插入到上下文中
4. 确保后续开发符合项目规范

**规范检查要点**:
- 代码风格和命名规范
- API 设计规范
- 项目结构规范
- 测试规范

**输出进度**:
```
实现进度：
[✓] 回忆项目规范
[→] 项目确认
[ ] 相关性探索
[ ] 需求澄清
[ ] 需求确认
[ ] 方案设计
[ ] 方案选择
[ ] 代码实现
[ ] 完成记录
```

## 阶段 0.5: 项目确认

**目标**: 确认当前操作的项目，为后续相关性探索提供准确的项目名称

**流程**:
1. 调用 `project_list()` 获取所有已注册项目
2. 根据当前工作目录或用户输入，确定目标项目
3. 保存 `project_id` 和 `project_name`，供后续阶段使用

**确认方式**:
- **方式 1（自动匹配）**: 如果当前工作目录匹配某个项目的 path，自动选择
- **方式 2（用户选择）**: 如果有多个候选项目，展示列表供用户选择
- **方式 3（用户输入）**: 如果没有匹配项目，提示用户输入项目名称或创建新项目

**输出示例**:
```
项目确认：
- 当前项目: <项目名称> (ID: <project_id>)
- 项目路径: <项目路径>
```

**如果项目不存在**:
- 询问用户是否需要注册新项目
- 调用 `project_register()` 注册项目

**输出进度**:
```
实现进度：
[✓] 回忆项目规范
```

## 阶段 1: 相关性探索

**目标**: 在提问前，查询过往记录，了解是否有相似功能/笔记

**流程**:
1. **调用 memory-query 技能**（推荐）：
   - 使用阶段 0.5 确认的 `project_name` 作为第一个参数
   - 使用功能相关关键词作为第二个参数
   - 示例：`Skill: memory-query, args: "<project_name> <搜索关键词>"`
2. 或者**直接使用 Memory MCP** 查询：
   - 使用阶段 0.5 确认的 `project_id`
   - 查询关键词：
     - 功能相关的关键词（如"用户""登录""API"等）
     - 模块相关的标签（如"auth""ui""backend"等）
3. 分析查询结果，提取有价值信息：
   - 相似功能的实现方案
   - 相关的架构决策
   - 历史问题或注意事项

**查询示例（使用 memory-query 技能）**:
```
# 阶段 0.5 已确认 project_id 和 project_name
# 调用 memory-query 技能，传入项目名称和搜索关键词
Skill: memory-query, args: "<project_name> <搜索关键词>"
```

**查询示例（直接使用 Memory MCP）**:
```
# 使用阶段 0.5 确认的 project_id
# 1. 查询 features 分组（按功能关键词）
project_get(
    project_id="<已确认的project_id>",
    group_name="features"
)

# 2. 查询 notes 分组（查找实现方案）
project_get(
    project_id="<已确认的project_id>",
    group_name="notes"
)

# 3. 分析结果，寻找与当前需求相关的记录
```

**相似记录判断标准**:
- 功能名称/关键词相似度 > 70%
- 功能描述的核心概念一致
- 涉及相同的模块或技术栈
- 示例：已有"用户登录功能"，新需求"用户登录优化" → 视为相似

**探索结果处理**:
- 如果找到相似记录：记录 `similar_feature_id` 和 `similar_note_id`，在后续阶段中复用
- 如果未找到相似记录：标记为全新功能，后续创建新记录

**输出示例**:
```

  - 状态: <completed/in_progress>
```

```
[✓] 相关性探索
```

## 阶段 2: 需求澄清

**目标**: 基于相关性探索结果，通过 1-3 轮精准提问，明确功能需求细节

**流程**:
1. 分析用户提供的功能描述
2. 结合相关性探索结果，识别：
   - 已明确的事项（跳过）
   - 仍需澄清的模糊点
   - 边界条件、依赖关系
3. 每轮提问 2-4 个**针对性**问题
4. 用户回答后，判断是否需要继续澄清

**提问原则**:
- **优先利用历史信息**: 如果相似记录中已有明确信息，不再重复询问
- **聚焦差异点**: 针对当前需求与已有记录的差异进行提问
- **渐进式深入**: 先问核心问题，再问细节

**提问示例（有相似记录）**:
```
需求澄清 (第 1/3 轮):

参考过往记录："用户登录功能" 已实现基础登录流程。

针对当前"用户登录优化"需求，需要澄清：
1. 本次优化主要针对哪个方面？（性能/安全/体验）
2. 当前实现中具体存在什么问题？
3. 预期优化后的效果指标是什么？
```

**提问示例（无相似记录）**:
```
需求澄清 (第 1/3 轮):
1. 这个功能的触发条件是什么？（用户操作/定时任务/API调用）
2. 输入数据格式是怎样的？
3. 预期的输出结果是什么？
4. 是否有性能或安全方面的要求？
```

**澄清完成标准**:
- 功能边界清晰
- 输入输出明确
- 技术约束确认

用户确认后，调用 Memory MCP：

```

project_list()

# 2. 【重要】复用或创建 Feature 记录
# 阶段1已进行相关性探索，此时已有探索结果：
#   - 如果发现相似记录：
#     - 询问用户是否复用该记录（展示已有记录的 content 和状态）
#     - 用户确认复用：使用已有 feature_id，跳过创建步骤
#     - 用户不复用：继续创建新记录
```

### 5.3 记录 Note

将方案和测试用例记录为 Note：

```
# 【重要】首先检查 Feature 是否已有关联的 Note
# 获取当前 Feature 详情，查看 note_id 字段
current_feature = project_get(
    project_id="<项目ID>",
    group_name="features",
    item_id="<feature_id>"
)
existing_note_id = current_feature.get("note_id")

if existing_note_id:
    # Feature 已有关联 Note
    existing_note = project_get(
        project_id="<项目ID>",
        group_name="notes",
        item_id=existing_note_id
    )
    # 询问用户：
    # - 选项1：更新现有 Note (推荐)
    # - 选项2：创建新 Note (会替换关联)

    if 用户选择更新现有:
        project_update(
            project_id="<项目ID>",
            group="notes",
            item_id=existing_note_id,
            content="<追加新方案和测试用例>",
            description="<更新后的描述>",
            tags="<追加标签>"
        )
        note_id = existing_note_id
    else:
        # 创建新 Note
        project_add(...)
        note_id = 新note_id
else:
    # Feature 无关联 Note，检查是否有相似 Note
    project_get(
        project_id="<项目ID>",
        group_name="notes"
    )
    # 分析已有 notes 列表，检查是否有与当前功能相关的实现方案记录
    # 如果找到相似记录，询问用户是否复用

    # 创建新 Note
    project_add(...)
    note_id = 新note_id

# 关联 Feature 和 Note（如果 note_id 变化了）
project_update(
    project_id="<项目ID>",
    group="features",
    item_id="<feature_id>",
    note_id=note_id
)
```

**Note 处理原则**:
1. **优先更新现有**: 如果 Feature 已有关联 Note，优先更新而不是新建
2. **保持历史**: 追加内容而不是替换，保留历史实现记录
3. **清晰分隔**: 使用 `---` 或日期标记分隔不同的实现版本

**相似 Note 判断标准**:
- description 中包含相同的功能名称
- tags 中有相同的功能相关标签
- 示例：已有"用户登录 - 实现方案"，新需求"用户登录优化" → 可更新同一 Note

## 阶段 6: Git 分支准备

**目标**: 在代码实现前准备 Git 开发分支

**流程**:
1. 此时已有 `feature_id`（新建的或已存在的）
2. 调用 `git-branch` 技能准备分支：
   ```
   Skill: git-branch, args: "prepare <feature_id> feature"
   ```
3. 技能会：
   - 检查当前分支是否为 main/master（如果是则拒绝）

## 标签处理流程

```
1. 调用 project_tags_info(project_id) 获取已注册标签
2. 根据功能类型匹配合适的已注册标签
3. 若无合适标签:
   - 调用 tag_register() 注册新标签
   - 标签名称使用英文，描述10-50字
4. 使用选定的标签记录 Feature/Note
```

**常用标签示例**:
| 标签名 | 语义描述 |
|--------|----------|
| feature | 新功能开发 |
| api | API 相关功能 |
| ui | 用户界面相关 |
| core | 核心模块 |
| implementation | 实现细节 |
| high | 高优先级 |
| medium | 中优先级 |
| low | 低优先级 |

## 异常处理

| 场景 | 处理方式 |
|------|----------|
| 用户取消流程 | 询问是否保留已记录的 Feature |
| 方案设计阻塞 | 提供简化方案或请求用户指导 |
| 实现过程中发现问题 | 更新 Note 记录，保持 Feature 状态为 in_progress |
| 测试失败 | 记录问题到 Note，继续修复 |
